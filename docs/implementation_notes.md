# 实现细节说明

这个文档用来解释项目里的关键设计决策，帮助你真正理解这个模板为什么这样写。

## 1. 配置层

配置统一收敛到 `core/config.py`，原因如下：

- 本地可能使用 Milvus Lite，测试环境可能是独立 Milvus，线上可能是托管服务。
- 集合名、维度、度量方式都属于运行参数，不应该硬编码在业务逻辑里。
- 统一配置类后，任何模块都能拿到同一份配置，避免多处定义不一致。

## 2. 集合模型

这个模板使用一张集合承载文档数据，包含如下逻辑字段：

- `doc_id`：主键，业务主键可读性强，方便排查问题。
- `embedding`：向量字段，Milvus 检索核心。
- `text`：原始文本，便于返回给上层业务。
- `source`：来源字段，便于做过滤和追踪。
- `tags`：标签列表，便于做简单分类。
- `metadata`：扩展字段，便于承载业务属性。
- `created_at` / `updated_at`：时间信息，便于审计和调试。

其中非主键和非向量字段以动态字段方式保存，这样模板复杂度更低，也更方便你继续扩展。

## 3. 为什么业务层不直接接收向量

当前 API 直接接收文本并在服务端完成向量化，而不是要求调用方自己传 embedding，原因是：

- 对多数业务来说，调用方只关心文本，不想自己处理 embedding。
- 统一由服务端生成向量，能保证维度、模型版本、归一化策略一致。
- 后续替换 embedding 实现时，不需要所有调用方一起修改接口。

如果你的场景是“离线生成 embedding，再写入 Milvus”，可以再增加一个只接收向量的内部接口。

## 4. 为什么使用轻量 embedding

模板使用 `DemoTextEmbedding`，主要为了：

- 尽量少依赖外部模型资源。
- 让示例重点聚焦在 Milvus 的接入方式和工程结构。
- 提供一个稳定、可测试、可替换的接口边界。

这不意味着生产环境应该继续使用它。生产环境应替换成真实语义向量模型。

## 5. 过滤表达式

搜索接口支持按 `source` 过滤，本质上是为了展示 Milvus 的“向量相似度 + 标量条件”混合查询能力。

真实场景中，你通常会把过滤条件扩展成：

- `tenant_id`
- `knowledge_base_id`
- `document_type`
- `status`
- 时间范围

这样才能真正支撑企业级检索系统。

## 6. 关于索引

这个模板为了通用性，没有强行写死复杂索引参数，而是优先保证：

- 结构清晰
- 可以本地跑通
- 可以直接切远程服务

当你进入生产阶段后，应根据数据规模和延迟目标评估：

- `AUTOINDEX`
- `HNSW`
- `IVF_FLAT`
- `IVF_PQ`

以及对应参数，例如 `M`、`efConstruction`、`nlist`、`nprobe` 等。

## 7. 推荐扩展顺序

如果你要把模板升级成真实业务系统，建议按下面顺序推进：

1. 先替换 embedding
2. 再补充切片和文档导入流程
3. 再加入多租户过滤
4. 最后再优化索引和性能

这样迭代路径更平滑，也更不容易一下子把工程复杂度拉得太高。
