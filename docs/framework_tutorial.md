# FastAPI + Milvus 框架教程

这份文档专门讲“这个项目的框架是怎么搭起来的”，方便你理解，而不是只会运行。

## 1. 先理解这个项目的定位

这个项目本质上不是一个单独脚本，而是一个标准的后端服务。

它解决的是下面这条链路：

1. 用户发送文本
2. 服务把文本转成向量
3. 服务把向量和元数据写入 Milvus
4. 用户再发送查询文本
5. 服务把查询文本转成向量
6. Milvus 根据向量相似度返回结果
7. FastAPI 把结果整理成 JSON 返回

所以它本质上是一个“向量检索服务后端”。

## 2. 为什么用 FastAPI

FastAPI 很适合这种项目，原因很现实：

- 它天然就是做 API 服务的。
- 参数校验和接口文档几乎是现成的。
- 很适合把“配置层、业务层、数据库层、路由层”拆开。
- 后续加登录、日志、监控、RAG、任务队列都比较自然。

如果只是做一个临时脚本，纯 Python 也可以。
但如果目标是“可落地的项目”，FastAPI 更合适。

## 3. 整个框架怎么分层

这个项目采用的是常见的后端分层思想。

### 3.1 `main.py`

`main.py` 是应用入口。

它负责：

- 创建 FastAPI 应用
- 在启动时初始化 Milvus
- 把 service 挂到 `app.state`
- 注册路由

你可以把它理解成整个项目的“装配中心”。

### 3.2 `core/config.py`

这个文件负责统一读取环境变量。

为什么必须单独抽出来：

- 本地环境和生产环境配置不同
- Milvus 地址、集合名、向量维度都可能变化
- 如果把这些值写死在代码里，后期会很难维护

所以规范做法就是：配置和业务逻辑分离。

### 3.3 `db/milvus_client.py`

这一层专门处理 Milvus 连接与集合初始化。

它负责：

- 创建 `MilvusClient`
- 判断集合是否存在
- 必要时自动创建集合
- 处理本地 `Milvus Lite` 的数据目录

为什么不把这些逻辑直接写在路由里？

- 因为路由层应该尽量只关心请求和响应
- 数据库连接和集合初始化属于基础设施层职责

### 3.4 `schemas/document.py`

这里定义 API 的数据模型。

例如：

- 请求里允许哪些字段
- 哪些字段必填
- 返回值长什么样

这层的作用很重要，因为它让接口边界变清晰。

### 3.5 `services/vector_service.py`

这是业务核心层。

它负责：

- 接收文本
- 调用 embedding 组件生成向量
- 调用 Milvus 写入或检索
- 把 Milvus 原始结果整理成更稳定的业务响应

这是整个项目最关键的一层。

简单说：

- `api/` 负责“收请求”
- `services/` 负责“做事情”
- `db/` 负责“连数据库”

### 3.6 `api/routes.py`

这是路由层。

它负责：

- 定义 URL
- 绑定 HTTP 方法
- 接收请求模型
- 调用 service
- 返回响应

它不应该直接写数据库逻辑。

如果路由层直接写 Milvus 调用，项目一大就会越来越乱。

### 3.7 `utils/demo_embedding.py`

这个文件是示例向量器。

它的作用不是提供最强语义效果，而是为了：

- 让项目零额外模型依赖就能跑通
- 明确展示“文本 -> 向量”这一步
- 给未来替换真实模型留出清晰接口

后面你完全可以把它换成：

- OpenAI embedding
- BGE / M3E
- sentence-transformers
- 公司内部模型服务

## 4. 一个请求是如何流动的

以“搜索接口”为例，请求流转如下：

1. 用户请求 `/api/v1/documents/search`
2. `api/routes.py` 接收请求
3. 路由调用 `VectorDocumentService.search_documents()`
4. service 先把查询文本转成向量
5. service 调用 `MilvusClient.search()`
6. Milvus 返回检索结果
7. service 把结果转成统一响应模型
8. FastAPI 返回 JSON

这个流程很好理解，也方便调试。

以后如果某一步出问题，你很容易定位：

- 入参错了，看 `schemas/`
- 路由错了，看 `api/`
- 业务逻辑错了，看 `services/`
- 数据库连接或集合问题，看 `db/`
- 配置错了，看 `core/`

## 5. 为什么这个结构适合后续扩展

因为它天然支持演进。

比如你以后想加：

- 文档切片
- 批量导入
- 多租户隔离
- 鉴权
- 日志
- RAG 召回
- 重排模型

都可以在现有结构上自然生长，不需要推倒重来。

## 6. Milvus 在这里扮演什么角色

Milvus 在这个项目里就是“向量数据库层”。

它负责：

- 存储向量
- 存储与向量关联的标量字段
- 基于向量相似度做检索
- 支持结合标量条件做过滤

在这个项目里，Milvus 不是单独直接面向用户的，而是由 FastAPI 包了一层业务接口。

这也是企业项目里最常见的做法。

## 7. 为什么默认用 Milvus Lite

因为学习和本地开发阶段，最需要的是：

- 快速启动
- 少依赖
- 先跑通流程

Milvus Lite 非常适合这个阶段。

当你以后切到真实部署环境，再把 `.env` 改成远程地址即可，不需要重写业务代码。

## 8. 你应该重点掌握哪些文件

如果你是第一次接触这类项目，我建议按这个顺序读：

1. `README.md`
2. `main.py`
3. `api/routes.py`
4. `services/vector_service.py`
5. `db/milvus_client.py`
6. `core/config.py`
7. `schemas/document.py`
8. `utils/demo_embedding.py`

这个顺序最容易从整体看到细节。

## 9. 你下一步可以怎么学

建议这样学：

1. 先跑通项目
2. 用 Swagger 调一次接口
3. 看 `main.py` 理解启动流程
4. 看 `routes.py` 理解接口定义
5. 看 `vector_service.py` 理解业务逻辑
6. 看 `milvus_client.py` 理解数据库初始化
7. 最后再替换 embedding

这样不会一上来就被细节压垮。

## 10. 后续升级路线

如果你想把这个项目继续做强，建议按这个顺序扩展：

1. 接入真实 embedding 模型
2. 增加文档切片与批量导入
3. 增加租户字段和过滤条件
4. 增加 Docker Compose
5. 增加日志、鉴权、监控
6. 扩展成完整 RAG 检索后端

这个顺序会比较平滑，也最符合实际项目演进路径。
